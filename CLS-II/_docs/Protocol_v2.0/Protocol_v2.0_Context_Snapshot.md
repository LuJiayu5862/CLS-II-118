## 一、为什么需要 v2.0？（设计动机）

### 现状 v1.1 的问题

TcLCS-UDP v1.1 是**硬编码式**协议：每次项目需要读写的变量发生变化，就必须在**主站（ST/C 语言）**和**上位机（C#）**两端同步修改报文定义、重新编译部署。这违反了 DRY 原则，导致：
- 每个新项目都要重复开发通信层
- 上位机 C# 代码无法跨项目复用
- 维护成本高，容易出现两端不一致的 Bug

### v2.0 的核心目标

| # | 目标 | 说明 |
|---|------|------|
| 1 | 上位机免重编译 | 通过 `.json` 配置文件驱动读写变量与 UI 绑定，最多修改配置文件即可适配新项目 |
| 2 | 主站动态注册 | 主站通过 `PARAM_ENTRY` 结构体数组注册需要暴露的变量（含名称、类型、权限、分组等元信息）|
| 3 | 单变量 & 分组批量读写 | 支持按 `ParamID` 批量读写，也支持按 `GroupID` 整组读写，减少命令往返开销 |
| 4 | 高频实时性 | 运行阶段仅传输 `ParamID + Value`，理论支持 ≤ 2ms 通信周期（与 v1.1 实测性能一致）|
| 5 | 高精度示波器 | 主站内置 Trace Buffer（最小 250μs 采样精度），上位机下发配置后批量取回，不依赖通信实时性 |

## 二、v2.0 通信模型（两阶段）

```
阶段一：握手 / 枚举（启动时一次）
  上位机 ──GET_PARAM_DICT(0x10)──▶ 主站
  上位机 ◀──PARAM_DICT_RESP(0x90)── 主站  （分页，直到 LAST_FRAG=1）
  上位机建立内存中的 ParamDictionary

阶段二：运行期循环
  ┌─ 2ms 周期写  ──WRITE_BY_ID(0x21) / WRITE_GROUP(0x23)──▶ 主站
  ├─ 差分写      ──WRITE_BY_ID(0x21)──▶ 主站（检测到变化入队）
  ├─ 2ms 周期读  ──READ_GROUP(0x22)──▶ 主站（实时数据组）
  ├─ 100ms/1s 读 ──READ_BY_ID(0x20)──▶ 主站（非实时数据）
  └─ Trace 示波器 ──TRACE_CONFIG(0x30)→TRACE_START(0x31)→TRACE_UPLOAD(0x34)──
```

## 三、与 v1.1 的关系

| 项目 | v1.1 | v2.0 |
|------|------|------|
| Version 字段 | `0x02` | `0x03` |
| Header 长度 | 11B | 12B（+Reserved 2B）|
| 端口 | 5050 (Param) / 15000 (Data) | 新增 **5051**（v2.0 专用）|
| SOF / EOF / CRC | 0xAA 0x55 / 0x55 / CRC-16/MODBUS | **完全沿用** |
| CMD 范围 | 0x01~0x05, 0xEE | 0x01~0x05 兼容 + **0x10~0x3F 新增** |
| 读写方式 | 按 SubID 硬编码 | 按 **ParamID 动态寻址** |
| 变量元信息 | 无 | **PARAM_ENTRY 结构体**（名称/类型/权限/分组/周期类）|
| 示波器 | 无 | **Trace Buffer**（主站内置，250μs 精度）|

**双版本共存策略**：主站在端口 5051 上收到帧后，先检查 `Version` 字段：
- `0x02` → 转发给 v1.1 处理逻辑（`FB_RequestHandler`）
- `0x03` → 转发给 v2.0 处理逻辑（`FB_ParamServer`，待实现）

## 六、关键设计决策记录（ADR）

### ADR-V2-001：不使用 Modbus，继续扩展自有 UDP 协议
- **动机**：用户希望实现类似 Modbus 的通用化参数化通信，但 Modbus 寄存器模型不支持变量名/类型/权限等元信息，且严格的请求-应答串行模型对高频写不友好。
- **决策**：在现有 TcLCS-UDP 基础上扩展 v2.0，引入 `PARAM_ENTRY` 自描述机制，保留所有已验证的实时性特性。
- **参考**：Modbus TCP 局域网 RTT 理论 <1ms，但 Windows 轮询抖动和无自描述能力是根本限制。

### ADR-V2-002：示波器采用 Trace Buffer 模型而非实时推送
- **动机**：工业驱动器（西门子 SINAMICS 等）的上位机示波器并非依赖通信实时性，而是驱动器内部硬实时采样后批量上传。
- **决策**：主站内置 Trace Buffer（最小 250μs，与 PLC 任务周期一致），上位机仅需下发配置+取回数据，通信层无需承担高频实时性。
- **好处**：示波器精度达到 PLC 任务周期级别，完全解耦通信延迟。

### ADR-V2-003：v2.0 使用独立端口 5051
- **动机**：避免与 v1.1 的 5050/15000 端口冲突，支持双版本并行运行，便于渐进迁移。
- **决策**：v2.0 使用 UDP 端口 5051，v1.1 端口不变。

### ADR-V2-004：Header 扩展 Reserved 2B
- **动机**：为未来扩展（如 SubCMD、SessionID、多设备路由）预留空间，代价极小（+1B 对比 v1.1）。
- **决策**：当前填 0x0000，后续按需启用。
